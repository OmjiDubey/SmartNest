# SmartNest Frontend Integration Guide

## Purpose

> **Note on Scope**: This guide documents the REST APIs and Socket.IO payloads exposed by the backend. These payloads are the frontend contract and may differ from the raw MQTT payloads published by the ESP32 firmware because the backend intentionally normalizes and translates data into frontend-friendly models.

This document provides a comprehensive guide for frontend engineers (specifically React Native developers) on integrating with the SmartNest backend system. It covers REST API endpoints, authentication flows, real-time Socket.IO integration, data models, and common error handling patterns.

---

## 1. Project Overview

SmartNest uses a hybrid communication architecture:
- **REST APIs**: For explicit commands (e.g., turning on a relay, login/logout, fetching historical data) which return immediate acknowledgments (`cmd_id`).
- **Socket.IO**: For real-time telemetry and state changes (e.g., live sensor data, status updates, and command acknowledgements).

---

## 2. Base URL

- **REST API Base URL**: `http://<SERVER_IP>:<PORT>`
- **Socket.IO Base URL**: `http://<SERVER_IP>:<PORT>`

*(Replace `<SERVER_IP>` and `<PORT>` with your respective staging or production environment values)*

## 3. Backend Translation Layer & Naming Conventions

The backend clearly separates the MQTT contract from the frontend contract and acts as a translation layer.

**MQTT payloads** (from firmware) strictly use `snake_case`:
```json
{
  "device_id": "...",
  "master_lock": false,
  "digital_switch": false
}
```

**Frontend payloads** (exposed via REST and Socket.IO) are normalized into **`camelCase`** DTOs (Data Transfer Objects):
```json
{
  "deviceId": "...",
  "masterLock": false,
  "digitalSwitch": false
}
```

**Exception for Protocol Identifiers**:
The field `cmd_id` intentionally remains `snake_case` in frontend payloads. This is a core protocol identifier shared identically across REST responses, Socket.IO acknowledgements, MQTT acknowledgements, and internal command tracking. Keeping it identical avoids unnecessary translation overhead.

## 5. Timestamp Format

Timestamps are strictly defined based on their origin:

- **Backend-generated timestamps** (e.g., `lastUpdated`): Generated by the backend when it receives state updates. Always emitted as **ISO-8601 UTC strings** (e.g., `"2024-02-18T10:30:05.000Z"`).
- **Firmware-generated timestamps** (e.g., `timestamp` inside `command:ack`): Passed through directly from the hardware firmware. Always emitted as a **Unix epoch timestamp in seconds** (e.g., `1780000000`).

Pay close attention to the specific payload documentation to know which format to expect.

## 6. Authentication

The backend uses JWT (JSON Web Tokens) with a short-lived Access Token and a long-lived Refresh Token. 

### POST `/api/auth/login`
- **Description**: Authenticate user and get tokens.
- **Required Headers**: `Content-Type: application/json`
- **Request Body**:
  ```json
  {
    "username": "admin",
    "password": "password123"
  }
  ```
- **Success Response** (`200 OK`):
  ```json
  {
    "accessToken": "eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...",
    "refreshToken": "random_refresh_token_string"
  }
  ```
- **Error Responses**: `400 Bad Request`, `401 Unauthorized`

### POST `/api/auth/refresh`
- **Description**: Refresh an expired access token using the refresh token.
- **Required Headers**: `Content-Type: application/json`
- **Request Body**:
  ```json
  {
    "refreshToken": "random_refresh_token_string"
  }
  ```
- **Success Response** (`200 OK`):
  ```json
  {
    "accessToken": "eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9..."
  }
  ```

### POST `/api/auth/logout`
- **Description**: Invalidate the refresh token and log the user out.
- **Required Headers**: 
  - `Content-Type: application/json`
  - `Authorization: Bearer <accessToken>`
- **Request Body**:
  ```json
  {
    "refreshToken": "random_refresh_token_string"
  }
  ```
- **Success Response**: `204 No Content`

## 7. REST API Reference (Device Operations)

All device endpoints require the access token to be passed in the Authorization header:
`Authorization: Bearer <accessToken>`

### GET `/api/device/:deviceId/relays`
- **Description**: Get the current in-memory state of all relays for a given device.
- **Success Response** (`200 OK`):
  ```json
  {
    "success": true,
    "data": {
      "states": [false, true, false, false, false, false, false],
      "locks": [false, false, false, false, false, false, false],
      "masterLock": false,
      "digitalSwitch": false,
      "runtimeSec": [0, 3600, 0, 0, 0, 0, 0]
    }
  }
  ```

### POST `/api/device/:deviceId/relays/:relayNo`
- **Description**: Set a specific relay (1-7) to ON (`true`) or OFF (`false`).
- **Request Body**:
  ```json
  {
    "state": true
  }
  ```
- **Success Response** (`202 Accepted`):
  ```json
  {
    "success": true,
    "cmd_id": "cmd-162788349283-abc1",
    "status": "pending"
  }
  ```

### POST `/api/device/:deviceId/relays/:relayNo/toggle`
- **Description**: Toggle a specific relay.
- **Success Response** (`202 Accepted`):
  ```json
  {
    "success": true,
    "data": {
      "deviceId": "SmartNest_001",
      "cmd_id": "cmd-162788349283-abc1",
      "status": "pending"
    }
  }
  ```

### POST `/api/device/:deviceId/relays/:relayNo/lock`
- **Description**: Lock or unlock a relay so its state cannot be changed locally.
- **Request Body**:
  ```json
  {
    "locked": true
  }
  ```
- **Success Response** (`202 Accepted`)

### POST `/api/device/:deviceId/master-lock`
- **Description**: Set the master lock state for all relays.
- **Request Body**:
  ```json
  {
    "state": true
  }
  ```
- **Success Response** (`202 Accepted`)

### POST `/api/device/:deviceId/lights`
- **Description**: Turn all lights on or off at once.
- **Request Body**:
  ```json
  {
    "state": true
  }
  ```
- **Success Response** (`202 Accepted`)

### POST `/api/device/:deviceId/relays/off`
- **Description**: Turn off all relays on the device.
- **Success Response** (`202 Accepted`)

### POST `/api/device/:deviceId/relays/unlock`
- **Description**: Unlock all locked relays on the device.
- **Success Response** (`202 Accepted`)

### POST `/api/device/:deviceId/ac`
- **Description**: Control Air Conditioner properties. **Exactly one** of the following parameters must be provided per request.
- **Request Body Options**:
  ```json
  { "power": true }
  ```
  ```json
  { "temp": 24 } // Between 16 and 30
  ```
  ```json
  { "tempStep": "up" } // or "down"
  ```
  ```json
  { "fan": "auto" } // auto, min, low, med, high, max
  ```
- **Success Response** (`202 Accepted`)

### POST `/api/device/:deviceId/slave-reboot`
- **Description**: Reboot a slave microcontroller (digital or pzem).
- **Request Body**:
  ```json
  {
    "target": "digital" // or "pzem"
  }
  ```
- **Success Response** (`202 Accepted`)

### POST `/api/device/:deviceId/pzem/reset-energy`
- **Description**: Reset energy counters on the PZEM sensor.
- **Success Response** (`202 Accepted`)

### POST `/api/device/:deviceId/system/reboot`
- **Description**: Reboot the entire master device (ESP32).
- **Success Response** (`202 Accepted`)

## 8. Socket.IO Integration

To receive real-time state updates, the frontend must connect to the Socket.IO server.

### Connecting
Clients must authenticate by providing the JWT access token during connection:

```javascript
import { io } from "socket.io-client";

const socket = io("http://<SERVER_IP>:<PORT>", {
  auth: {
    token: "eyJhbGciOiJIUzI1NiIsInR5cCI6..."
  }
});
```

### Subscribing to a Device
Once connected, you must explicitly tell the server which device you want to listen to. The backend uses rooms based on the `deviceId`.

```javascript
// To subscribe and start receiving updates
socket.emit("subscribe", "SmartNest_001");

// To unsubscribe
socket.emit("unsubscribe", "SmartNest_001");
```

### Socket Lifecycle

Socket.IO handles **automatic reconnection** if the network drops. However, room subscriptions on the backend are tied to the active socket connection and are lost during a disconnection. 
**The frontend must emit `subscribe` again after every reconnection.**

| Event | Description | Frontend Action |
|--------|-------------|----------------|
| `connect` | Connected to backend | Subscribe to current device |
| `disconnect` | Connection lost | Show offline/reconnecting indicator |
| `reconnect` | Connection restored | Re-subscribe to the current device |
| `reconnect_error` | Reconnection attempt failed | Continue retrying and notify the user if appropriate |

```javascript
socket.on("connect", () => {
  // Always (re)subscribe upon connection
  if (currentDeviceId) {
    socket.emit("subscribe", currentDeviceId);
  }
});

socket.on("disconnect", (reason) => {
  console.log("Socket disconnected:", reason);
  // UI should show "Offline" or "Reconnecting..." state
});

socket.on("reconnect", (attempt) => {
  console.log(`Reconnected after ${attempt} attempts`);
  if (currentDeviceId) {
    socket.emit("subscribe", currentDeviceId);
  }
});

socket.on("reconnect_error", (error) => {
  console.error("Reconnection failed:", error);
});
```

## 9. Socket Events

The server will emit the following events to subscribed clients. Listen for these to update the UI reactively.

### `device:relays`
- **Direction**: Server → Client
- **When it is emitted**: Event-driven (whenever a relay state, lock, or runtime changes).
- **What UI should update**: Toggle switches, relay status indicators, lock icons.

### `device:sensors`
- **Direction**: Server → Client
- **When it is emitted**: Periodically (usually every few seconds based on firmware telemetry).
- **What UI should update**: Dashboard cards, live gauges (voltage, current, power, temperature, humidity). *Do not overwrite user input fields with this data.*

### `device:status`
- **Direction**: Server → Client
- **When it is emitted**: Periodically (device heartbeat, wifi strength, uptime).
- **What UI should update**: Overall device health indicators, Wi-Fi signal bars, SD card storage usage.

### `device:slaves`
- **Direction**: Server → Client
- **When it is emitted**: Event-driven or periodically.
- **What UI should update**: Slave microcontroller status indicators (PZEM, Digital Board).

### `command:ack`
- **Direction**: Server → Client
- **When it is emitted**: Event-driven (when the hardware explicitly acknowledges a command the frontend sent).
- **What UI should update**: Clear pending/loading states (spinners) on action buttons, or show an error toast if `ok` is false.

### `subscribe` / `unsubscribe`
- **Direction**: Client → Server
- **Purpose**: Sent by the frontend to join or leave a specific device's real-time data room.

## 10. Common Data Models

These models represent the payloads you will receive via Socket.IO events (`device:status`, `device:sensors`, `device:relays`, `command:ack`).

### Missing or Unavailable Sensor Values
When a hardware component is disconnected or offline (e.g., DHT sensor failure, PZEM offline, or Digital slave offline), the backend ensures:
- **Previous value retained**: Metric fields (like temperature, voltage, or current) retain their last known value rather than dropping to zero or null.
- **Status flags updated**: The failure is indicated via explicit boolean flags in the `status` payload (e.g., `pzemOnline: false`, `dhtOk: false`). 
*The frontend should look at these flags to determine if the displayed metric is stale or unreliable.*

### Status Payload (`device:status`)
```json
{
  "uptime": 3600,
  "wifi": {
    "ssid": "MyHomeNetwork",
    "rssi": -65
  },
  "mqttStatus": 1,
  "sd": {
    "ok": true,
    "total": 16000000,
    "used": 4500000
  },
  "digitalOnline": true,
  "pzemOnline": true,
  "pzemHealth": true,
  "dhtOk": true,
  "lastUpdated": "2024-02-18T10:30:00.000Z"
}
```

### Sensors Payload (`device:sensors`)
```json
{
  "voltage": 230.5,
  "current": {
    "main": 1.2,
    "digital": 0.5,
    "ac": 0.0
  },
  "power": {
    "ac": 0.0
  },
  "energy": {
    "acKwh": 12.5,
    "mainKwh": 45.2,
    "digitalKwh": 8.1
  },
  "environment": {
    "temperatureC": 24.5,
    "humidityPct": 45
  },
  "lastUpdated": "2024-02-18T10:30:05.000Z"
}
```

### Relays Payload (`device:relays`)
```json
{
  "items": [
    {
      "relay": 1,
      "state": true,
      "locked": false,
      "runtimeSec": 120
    }
  ],
  "masterLock": false,
  "digitalSwitch": false,
  "lastUpdated": "2024-02-18T10:30:10.000Z"
}
```

### Command Acknowledgment Payload (`command:ack`)
```json
{
  "cmd_id": "cmd-162788349283-abc1",
  "command": "relay_set",
  "ok": true,
  "message": "relay 1 set to ON",
  "timestamp": 1780000000
}
```

**Field Descriptions**:
- **`cmd_id`** (String): Unique identifier of the command, matches the one returned by the REST API. Used to resolve pending states.
- **`command`** (String): The action that was executed (e.g., `relay_set`, `ac_set`).
- **`ok`** (Boolean): Indicates whether the command was successfully executed by the hardware.
- **`message`** (String): Human-readable message or error description from the firmware. Useful for toasts/alerts if `ok` is false.
- **`timestamp`** (Number): **Unix epoch timestamp in seconds** indicating exactly when the acknowledgment was generated by the firmware.

## 11. Frontend Integration Flow

### Command Lifecycle
When the user triggers an action (e.g., toggling a relay), follow this flow:
1. User presses a button.
2. Frontend sends the REST API request.
3. Backend immediately returns a `cmd_id`.
4. Frontend stores the pending `cmd_id` and sets the button to a "loading" state.
5. Hardware executes the command asynchronously.
6. Backend emits `command:ack` over Socket.IO indicating success/failure.
7. Backend emits updated device state (e.g., `device:relays`) over Socket.IO.
8. Frontend clears the loading state using the `command:ack`.

> **Important on Event Ordering**: `command:ack` and `device:relays` may arrive in either order. The frontend must not rely on one arriving before the other. Both events should be handled independently. The UI should always update its visual state using the actual device state events (like `device:relays`), rather than assuming the command succeeded.

### Handling a Relay Command (Example)
```javascript
// 1. Listen for Acks globally
socket.on("command:ack", (payload) => {
  if (payload.cmd_id === pendingCmdId) {
    if (payload.ok) {
      // Clear spinner (Success)
    } else {
      // Clear spinner and show error: payload.message
    }
  }
});

// 2. Trigger the action via REST API
const response = await api.post(`/api/device/${deviceId}/relays/1`, { state: true });
const pendingCmdId = response.data.cmd_id;
// Set UI to loading state for this action
```

## 12. Recommended Frontend Architecture

To ensure a robust integration, follow these architectural best practices:

- **One global Socket.IO connection**: Do not instantiate a new `socket.io-client` per screen. Initialize it once at the app level.
- **One shared Axios instance**: Use a centralized HTTP client instance configured with your Auth interceptors.
- **Global authentication state**: Manage login sessions globally so tokens are accessible by both Axios and Socket.IO.
- **Global device state (Redux / Zustand / Context)**: State updates via sockets should write to a central store so any mounted component can react instantly.
- **Global pending command tracking**: Track `cmd_id`s in a central map so responses can be resolved even if the user navigates away and back.
- **Avoid multiple socket connections**: Ensure cleanup on app unmount and do not manually call `.connect()` on every render.

## 13. Error Handling

| Situation | Frontend Action |
|-----------|----------------|
| Access token expired | Refresh access token automatically |
| Refresh failed | Logout user and redirect to login |
| Validation error (400) | Show validation message |
| Network timeout | Retry or notify user |
| Socket disconnected | Show offline banner |
| `command:ack` with `ok = false` | Display `payload.message` in a toast/alert |
| Internal server error (500) | Show generic "Something went wrong" error message |

## 14. Integration Checklist

- [ ] Implement JWT Auth flow (Login, Logout, Secure Storage).
- [ ] Implement Token Refresh interceptor in Axios/Fetch.
- [ ] Connect Socket.IO client using Auth Token.
- [ ] Re-emit `subscribe` on socket `connect` and `reconnect` events to restore room subscriptions.
- [ ] Build global Socket event listeners (`device:relays`, `device:sensors`, etc.) and connect them to state (Redux/Zustand/Context).
- [ ] Ensure UI relies on boolean status flags (e.g., `dhtOk`) to handle stale metrics.
- [ ] Implement UI loaders linked to `cmd_id` and resolve them via `command:ack` independently of state updates.
- [ ] Implement "Offline" or "Reconnecting" indicators for Socket drops.
